## 0. 项目定位

线下服装购物的「凑单 + 优惠券最优组合」计算器。用户录入**商品、必买标记、归属用户、优惠券规则**，App 算出**最省钱的实付方案**。

- 解决的真实问题：满减门槛凑单、多券组合（叠加 / 单用）、多人拼单分摊。
- 线下场景：手动录入，不爬电商 API。
- 三个 Tab（`MainScreen`）：**商品 / 优惠券 / 计算**。用户管理不新增 Tab，内嵌商品页左上角 Drawer。

## 3. 核心算法（`domain/algorithm/DiscountCalculator.kt`）

入口：单用户 / 合并解走 `calculateBestCombination(products, coupons, fillProducts, useFillProducts, multiUserMode, fairness)`。券先按 `isStackable == 当前模式` 过滤（一次仅一类券参与），再分流：
- `stackableCoupons` → `calculateStackableSolution`（每张独立整量算）
- `nonStackableCoupons` → `calculateNonStackableSolution`（分组背包）

### 3.1 叠加券（`isStackable = true`）
每张券基于**商品总价独立**算 `count = min(floor(totalAmount / threshold), usageLimit)`：
- 满减：`count` 次叠加，discount = `count × 面额`（总价 300 / 满 100 减 20 → count 3 减 60）。
- 折扣：达阈值 count=1，discount = 总价 × (折扣值 / 100)。
- 无门槛：count=1，discount = min(面额, 总价)。
- 多张叠加券各自算后累加 `totalDiscount`；`finalPrice = max(总价 − 总优惠, 0)`。
- **无分组、无凑单小物参与**（叠加券路径不调 `findBestExactGroups`）。

### 3.2 单用券（`isStackable = false`）
分组背包：每组配一张券，求总优惠最大的分组方案。
- **≤16 件精确解 `findBestExactGroups`**：
  1. `buildCandidates`：按券枚举所有「够门槛」的商品子集（bitmask 表示），带限界剪枝 DFS；凑单小物开启时补差价（gap ≤ 40），仅当 `discount > fillTotal`（净收益为正）才采用。
  2. `solve(availableMask, usagesLeft)`：记忆化递归，从剩余商品挑**不重叠**候选组，求总 `netSavings` 最大。
  3. `isBetterThan` 平局：netSavings > fillTotal > discount > 组数 > groupedTotal；单用户 `fairness = NONE`。
- **>16 件贪心解 `findGreedyGroups`**：商品升序，反复取面额最大券塞满门槛，牺牲最优性换性能。
- `calculateMultiCouponSolution` 同时支持叠加整量 + 单用分组，但 UI 当前按模式过滤只传一类。

### 3.3 多用户分摊（`CalculatorViewModel.calculateMultiUser`）
- `scoped` = 勾选商品 ∩ 勾选用户名下商品（必买始终保留）。
- `perUser` = 每人单独算（`multiUserMode = false`）。
- 两阶段分组：阶段1 每人自达标锁定单人组；阶段2 剩余跨用户凑单（`crossPool`, `multiUserMode = true`）。
- 分摊口径：
  - **叠加券**：增量 = `combined.totalDiscount − ΣperUser.totalDiscount`；按**余量占比**（`ownOriginal − 已用券门槛`）分摊增量。
  - **单用券**：逐组聚合，按**原价占比**分摊，凑单小物金额并入 `ownOriginal` 保证 `原价 − 优惠 = 应付`。
- 公平性 `FairnessStrategy`：仅总省钱平局时改优惠归属，不牺牲总省钱；「全局」候选必须用 `multiUserMode = false + FAIR`。

### 3.4 本轮修复：叠加券合并凑单算错（commit `b3235ee`）
- **根因**：`combined` 原用「各人单独用券 + 空 crossPool」拼，纯叠加券下 crossPool 空 → combined 优惠恒等于各人之和 → `incrementalDiscount` 恒 0、合并价偏高、人均分摊增量 0。
- **修复**：叠加券分支 `combined` 直接 `calculateBestCombination(scoped)` 对**合并总价整量重算**（`count = floor(合并总价 / 阈值)`）；单用券路径不变。
- **例**：A=150 B=150，满 100 减 20 叠加 → 修复前合并优惠 40 / 合并凑单 260（错）；修复后合并优惠 60 / 增量 20 / 各分 10 / 每人应付 120 / 合计 240 ✓

## 7. 关键设计决策 & 雷区（必看）

1. **Compose 手势坑**：`swipeable` 在 1.5.4 仍在 `material` 包；手势挂**前景 Card** 不挂外层 Box；`Switch` → `Checkbox`（避免吞拖拽）；左滑删除防误删。
2. **Drawer 误触**：`gesturesEnabled = false` 后遮罩点击关闭失效，必须有 X 关闭 + 菜单 toggle。
3. **叠加券合并必须整量重算**（§3.4 修复点），不能拼各人之和。
4. **公平性 = 纯 tie-break**，永不因公平少省钱。
5. **单用券 shares 必须含凑单小物金额**，否则 `原价 − 优惠 ≠ 应付`。
6. **余量占比用 `原价 − 已用券门槛`**（非单独算应付）。
7. **迁移保数据**：`MIGRATION_5_6` 自增 INSERT + 校正 `sqlite_sequence`，禁止指定 id。
8. **红色专用于必买**：`thresholdGapColor` 不得返回 `AppRed`。
9. **Compose 限制**：`remember` 不能置 `LazyListScope` content lambda 顶层；`Modifier.weight` 是 `RowScope` 扩展。
10. **不确定就问，别乱动手**（用户原话）。
