## 0. 一句话概括

一个**先自组、后凑单**的多用户凑单计算器：每人先用自己商品尽量用券（锁定为单人组），剩下来都不够门槛的人再跨用户凑单组；分摊口径按券模式区分——**单用券按原价占比，叠加券按余量占比**。

## 2. 核心锁定的业务规则（用户逐条拍板，不可擅自改）

| #   | 规则                     | 说明 / 来源                                                              |
| --- | ---------------------- | -------------------------------------------------------------------- |
| R1  | **单用券分摊 = 原价占比**       | 凑单组内每人按"本人商品原价 / 组内商品原价"分摊优惠                                         |
| R2  | **叠加券分摊 = 余量占比**       | 增量优惠按"原价 − 已用券门槛"的剩余金额占比分摊（非单独算应付）                                   |
| R3  | **单人已达标 → 标「单人组·自达标」** | 某人自己小计 ≥ 门槛，单独用券，不进凑单组，不显示分摊区                                        |
| R4  | **真凑单组判定**             | 候选组须同时满足：≥2 个不同 owner，且**每人各自小计 < 门槛**，且组总价 ≥ 门槛。任一 owner 已达标则该候选被剔除 |
| R5  | **先自组，后凑单（两阶段）**       | 阶段1 每人单独算并锁定已用券商品；阶段2 仅剩下来不够的人跨用户凑。绝不允许把某人已达标的大额商品拆去跟别人凑             |
| R6  | **分摊优惠金额 -¥XX 红 + 粗**  | 凑单组分摊区每人分到的优惠，13sp + Bold + `AppRed` 显示                              |
| R7  | **分组卡片类型标签**           | 单人组=蓝标签，凑单组=绿标签；未达标商品单独卡，橙/灰标签（`AppDangerSurface` 底）                 |
| R8  | **必买商品标红**             | `isRequired=true` 在计算页名称/价格用 `AppRed`，优先级最高，覆盖门槛色                    |
| R9  | **门槛提示橙绿**             | 距门槛 `≤20` 橙 `AppAmber`，`≤40` 绿 `AppGreen`（无红黄）                       |
| R10 | **汇总卡蓝底白字**            | `SummaryCard` 蓝底，实付款下方列出每人应付行（昵称白 / 原价−优惠白居中 / 应付浅蓝 `0xFF93C5FD`）    |
| R11 | **券模式互斥**              | 计算时只启用当前模式（叠加 or 单用）的券，两类券不混算（算法层支持混合但 UI 当前过滤）                      |

> 历史教训（务必尊重）：用户曾指出 49+59=108 同属一人却被拉进凑单组是严重 bug → 催生 R4/R5。

## 4. 算法层（`domain/algorithm/DiscountCalculator.kt`）

**已被修改，并非只读。** 现有签名：

```kotlin
fun calculateBestCombination(
    products: List<Product>,
    coupons: List<Coupon>,
    fillProducts: List<FillProduct> = emptyList(),
    useFillProducts: Boolean = false,
    multiUserMode: Boolean = false   // 新增：owner 感知分组
): BundleSolution
```

`multiUserMode` 贯穿整条链路：  
`calculateBestCombination` → `calculateNonStackableSolution` / `calculateMultiCouponSolution` → `findBestExactGroups` / `findGreedyGroups` → `buildCandidates`。

**owner 约束核心**（`buildCandidates`，行 ~389）：

```kotlin
if (multiUserMode) {
    val ownerTotals = group.groupBy { it.ownerId }
        .mapValues { (_, ps) -> ps.sumOf { it.originalPrice } }
    // 含 ≥2 个 owner，且任一 owner 小计已 ≥ 门槛 → 该候选不是真凑单，剔除
    if (ownerTotals.size >= 2 && ownerTotals.values.any { it >= coupon.threshold }) {
        continue
    }
}
```

- `perUser`（每人单独算）调用时 `multiUserMode` 默认 `false`（保持原行为）。
- 仅跨用户合并解（`combined` / `crossPool`）传 `multiUserMode = true`。
- **已知局限**：`findGreedyGroups`（商品 >16 件）路径的 owner 约束为轻量版，极端组合可能非最优；`<16` 件精确路径严格正确。日常凑单均在 <16 区间。

## 5. ViewModel（`CalculatorViewModel.kt` → `calculateMultiUser`）

触发：商品勾选/全选变化 `LaunchedEffect(selectedProductIds, ...)` 自动调用（行 112）。

核心流程（行 140–338）：

```
1. scoped = 勾选商品 ∩ 勾选用户名下商品（必买始终保留）
2. perUser = 每人单独算（multiUserMode=false）
3. 两阶段分组：
   阶段1：遍历 perUser，凡成功用券的商品 → 锁定为「单人组」(lockedProductIds)
   阶段2：剩余未锁定商品 → 若涉 ≥2 人且存在单用券 → crossPool = calculateBestCombination(..., multiUserMode=true)
4. combined = 单人组 + 跨用户凑单组 + 未达标商品 合并
5. 分摊 shares：
   - 叠加券(isStackable)：增量优惠 = combined.totalDiscount − ΣperUser.totalDiscount；
     按「余量占比」(ownerRemaining = ownOriginal − 已用券门槛) 分摊增量
   - 单用券：逐组聚合——
       · 单人组(locked)：该人全额享优惠 + 承担全部凑单小物费用
       · 凑单组(crossPool 中 ≥2 owner)：按原价占比分摊优惠 + 凑单小物按原价占比分摊成本
       · 跨用户池中的单人组(某人剩余刚够自己)：全额享优惠
       · 纯凑单小物组：费用均摊给所有人
       · 未达标商品：原价付
     ownOriginal 含分配到的凑单小物金额，保证「原价 − 优惠 = 应付」成立
6. 输出 _perUserSolutions / _combinedSolution / _shares / _incrementalDiscount
```

状态字段：`checkedUsers`、`perUserSolutions`、`combinedSolution`、`shares`、`incrementalDiscount`、`selectedProductIds`（用户手动勾选，跨导航持久化）。

`couponMode`（行 31）= `couponModeRepository.isStackableMode`，决定启用哪类券。

## 10. 雷区与历史踩坑（务必遵守）

1. **算法 owner 约束是多人模式专用**：`perUser` 调用必须 `multiUserMode=false`，只有合并解/crossPool 传 `true`，否则单人算会被错误剔除候选。
2. **先自组后凑单**：绝不在 ViewModel 把所有商品一股脑丢 `combined` 全局分组（曾导致把达标用户的大额商品拆去凑单、反而更贵）——必须走第 5 节两阶段。
3. **单用券 shares 必须含凑单小物金额**：`ownOriginal` 与 `userPayable` 都要把分配到的 `fillProducts` 金额算进去，否则「原价−优惠≠应付」。
4. **余量占比用 `原价 − 已用券门槛`**，不是单独算应付（R2 来源：benren 457−400=57 vs tfff 69−0=69，占比 45.2%/54.8%）。
5. **迁移保数据**：`MIGRATION_5_6` 自增 INSERT + 校正 `sqlite_sequence`，禁止指定 id。
6. **禁止删默认用户「本人」**；删其他用户同步清其商品。
7. **红色专用于必买**：`thresholdGapColor` 不得返回 `AppRed`（改 `AppAmber`/`AppGreen`）。
8. **Compose 限制**：`remember` 不能置于 `LazyListScope` content lambda 顶层（须提升到 LazyColumn 外）；`Modifier.weight` 是 RowScope 扩展，独立 Composable 内由调用方传 `modifier=Modifier.weight(1f)`。
9. **用户偏好**：改动前先出方案/设计稿确认再动手；删除字段只删指定文字，勿顺手大改。
