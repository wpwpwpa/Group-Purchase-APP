# 代码审查标准与流程（Code Review Standards & Process）

> 适用范围：本仓库（dealoptimizer，Kotlin + Jetpack Compose + Hilt + Room + MVVM）
> 制定日期：2026-07-12 ｜ 维护人：研发团队 ｜ 版本：v1.0
>
> 本文档的目标是**把「代码质量参差不齐」变成可量化、可执行的统一标尺**，而不是靠个人感觉。
> 所有条款都尽量用本仓库现有代码做正例 / 反例，方便团队对齐认知。

## 2. 架构分层约定（强制，违反即 🔴）

分层是本项目目前**做得好**的地方，必须守住：

```
presentation (ui / viewmodel / di)   ← 只负责渲染与编排，不写业务规则
        ↓ 调用
domain (algorithm / model)            ← 纯 Kotlin，无 Android 依赖，可单测
        ↑ 读取
data (repository / dao / model)       ← 只负责存取，不写业务规则
```

**强制规则：**
1. 🔴 **业务规则只能存在于 `domain` 层**。ViewModel、Composable、Repository 不得重新实现折扣/分摊/分组等算法。
   - 反例：`CalculatorViewModel` 在 `combined`（约 L270–274）和 `buildNonStackableShares`（约 L380–404）里**手抄了三套折扣公式**，与 `DiscountCalculator.calculateSingleUseDiscount` 逻辑重复。一旦折扣语义变更，这三处会悄悄分叉 → 算错钱。
   - 正例：`DiscountCalculator` 是纯函数、零 Android 依赖、可单测，全项目都应像它这样收敛领域逻辑。
2. 🔴 **`domain` 层不得 import `android.*` / `androidx.*`（除 `annotation` 外）**。发现即打回。
3. 🟡 `Repository` 是数据访问的唯一出口；Composable / ViewModel 不得直接持 `Dao` 引用。
4. 🟡 ViewModel 暴露给 UI 的应是**不可变** `StateFlow`（你现在 `_solution`/`solution` 的命名与封装是对的，保持）。

## 3. 代码审查标准（Checklist，分级 🔴 / 🟡 / 💭）

> 用法：评论时复制对应格式，写明**文件:行号 + 为什么 + 建议**。

### 3.1 🔴 阻断级（Blocker，必须修，否则禁止合并）

| 类别 | 标准 | 本项目真实反例 |
|------|------|----------------|
| 数据安全 | 禁止 `fallbackToDestructiveMigration()`——缺迁移会**静默清空用户数据** | `HiltModule.kt:40`、`AppContainer.kt:44` 都开着 |
| 主线程 | 禁止 `allowMainThreadQueries()`——ANR 风险 | `HiltModule.kt:29`、`AppContainer.kt:42` |
| 迁移校验 | `RoomDatabase` 必须 `exportSchema = true` 并导出 `schemas/` | `AppDatabase.kt:19` 设了 `false`，等于关掉迁移校验 |
| 领域泄漏 | 业务算法不得出现在 ViewModel/UI | 见 §2 反例 |
| 空安全 | 禁止 `!!` 与未防护的 `lateinit` 读取 | `AppContainer.kt:11` `lateinit var context` 可被误读抛异常 |
| 死代码 | 不得保留可编译但不可达、且带隐患的代码 | `AppContainer.kt` 整文件（内存已确认是死代码） |
| 正确性 | 同入口的不同代码路径必须产出一致结果 | `findGreedyGroups` 未传 `fairness`，`products.size>16` 时丢掉公平策略（见 §3.2） |

**阻断级评论格式：**
```
🔴 **[类别]：一句话结论**
文件:行号 — 问题描述。

**为什么：** 数据丢失 / ANR / 算错钱的具体后果。

**建议：** 具体改法（如删 `fallbackToDestructiveMigration()`，改用显式 `addMigrations` 全量注册）。
```

### 3.2 🟡 建议级（Suggestion，应修，需 reviewer 与作者达成共识）

| 类别 | 标准 | 本项目真实反例 / 说明 |
|------|------|----------------------|
| 一致性 | 算法不同分支/路径行为必须一致 | `DiscountCalculator.findGreedyGroups`（约 L329）签名**没有 `fairness` 参数**，而精确分支 `findBestExactGroups` 有。商品 >16 件时公平策略失效，且两条路径结果不可比 |
| 金额精度 | 货币用 `Double` 需全链路统一舍入与**带容差比较**；新增金额运算要集中 | 全局 `Double` 算钱（0.1+0.2≠0.3）；长期建议 `BigDecimal` 或 `Long` 分；至少统一比较用 `delta` |
| 错误处理 | 关键路径（计算/存取）要 try/catch 并暴露错误态，不能让协程静默失败 | `CalculatorViewModel` 各 `launch` 内 `.first()`/DB 调用无 try/catch，异常会被吞 |
| 重复 | 同一逻辑出现 ≥2 处应抽公共函数 | 折扣公式在 3 处重复（§2）；`"混合场景…去掉 isStackable 过滤即可"` 注释在 ViewModel 重复 3 次 |
| 命名 | 布尔/状态字段语义要明确；避免 `data`/`info` 之类模糊词 | 现有命名整体 OK，仅提醒新增时遵守 |
| 注释 | 注释解释 **why**，不解释 what；中文注释当前质量高，保持 | `DiscountCalculator` 的公平性/剪枝注释是范本，鼓励 |
| 测试 | 改动领域逻辑必须补/改对应单测 | 贪心分支丢 fairness 这类问题，现状测试未覆盖 `>16` 商品场景 |

### 3.3 💭 建议级微调（Nit，可选，不阻塞）

- 大函数拆分：`buildNonStackableShares`（约 80 行）可拆「单人组」「凑单组」「未达标」三段纯函数，便于单测。
- 重复默认参数：`useFillProducts: Boolean = fillProducts.isNotEmpty()` 在多个签名出现，属可控，可统一约定。
- 魔法数：如 `MAX_AUTO_FILL_GAP = 40.0` 已抽常量（好），新增阈值类数字请同样抽 `companion const`。
- 迁移里手改 `sqlite_sequence`（`MIGRATION_5_6`）是高难度操作，建议加注释说明「为什么必须」，目前已有，保持。

## 4. 代码审查流程（机制）

### 4.1 角色与职责

| 角色 | 职责 |
|------|------|
| **作者（Author）** | 提交前完成 §4.2 自查；PR 描述写清「改了什么 / 为什么 / 怎么测」；及时响应评论 |
| **审查人（Reviewer）** | 按 §3 逐项核对；至少 1 人；给出明确 Approve / Request Changes |
| **架构守护（可选，资深）** | 仅在涉及 `domain` 层接口、Room 迁移、DI 图变更时介入 |

> 小团队建议**交叉互审 + 关键改动双人审**。每个 PR 至少 1 个 Approval。

### 4.2 提交前自查清单（Author 必勾）

- [ ] 编译通过、`./gradlew lint` 与单测绿
- [ ] 无 `allowMainThreadQueries` / `fallbackToDestructiveMigration` / `exportSchema=false` 这类数据风险
- [ ] 业务算法只在 `domain` 层，ViewModel 直接复用 `BundleSolution` 而非重算
- [ ] 新增/修改的领域逻辑有对应单测（带浮点容差）
- [ ] 无死代码、无 `!!`、无未防护 `lateinit`
- [ ] 注释说清 why；无大片复制粘贴

### 4.3 Review 轮次与时效（SLA）

| 环节 | 时效 | 说明 |
|------|------|------|
| 作者提交 → 首个 reviewer 响应 | ≤ 1 个工作日 | 避免长时间挂起 |
| Reviewer 完成首轮 | ≤ 1 个工作日 | 大 PR 可拆分评审 |
| 作者修复 → 复审 | ≤ 1 个工作日 | 🔴 必须全部解决 |

**PR 体量建议**：单 PR 控制在「一个关注点」，≤ 400 行改动最佳；算法/迁移类大改请提前拆 commit。

### 4.4 分级处理规则（怎么处置评论）

- 🔴 **必须修**：作者修完、reviewer 确认后才能合并。
- 🟡 **应修**：作者要么修，要么在 PR 里写明「为何暂不改 + 后续 issue 单号」，由 reviewer 判定放行。
- 💭 **可选**：不阻塞合并；可记进技术债清单。

### 4.5 合并门槛（Merge Gate）

满足以下全部条件方可合并：
1. CI 绿（构建 + lint + 单测）。
2. 至少 **1 个 Approval**。
3. **0 个未解决的 🔴**。
4. 涉及 Room 版本号 / 迁移 / `domain` 接口变更时，必须有第二人确认。

### 4.6 工具链（建议逐步接入）

| 工具 | 作用 | 优先级 |
|------|------|--------|
| **ktlint** | 统一 Kotlin 风格（缩进、import 顺序） | 高，先用起来 |
| **detekt** | 静态坏味道（复杂度、重复、潜在 bug） | 高 |
| **Room schema 导出 + 校验** | 把 `exportSchema=true` 的 `schemas/` 提交进 Git | 高（直接防数据丢失） |
| **单元测试 CI 门禁** | 算法层单测必须全绿才让合并 | 高 |
| **架构守护测试（如 ArchUnit 思路）** | 断言 `domain` 不依赖 `android.*` | 中，可后续加 |
