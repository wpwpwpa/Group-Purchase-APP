---

## 1. 背景与问题

计算机用二进制存小数，`Double` 无法精确表示多数十进制小数。典型现象：

```
0.1 + 0.2  ≠ 0.3        → 实际得到 0.30000000000000004
```

本项目所有金额都用 `Double` 存储与运算。单笔误差极小（约 1e-15），但**优惠叠加、按人数占比分摊、合并凑单**会让误差累加，极端组合下可能出现「某人应付金额差 1 分」「各人分摊之和 ≠ 总优惠」的对账不平。

**当前实际风险**：客单价小、凑单常整，误差基本被「显示时四舍五入到分」吃掉，尚未收到算错反馈。但一旦上架收钱或做财务对账，这就是必须修的隐患。

### 2.5 ⚠️ 关键边界（改造前必须先厘清）
`ProductScreen.kt:308-319`：
```kotlin
val price = priceDigits.toPrice()          // toPrice() 返回 Int（整数「分」）
if (price > 0) {
    …
    originalPrice = price.toDouble()        // ← Int(分) 被强转成 Double 存入
}
```
- `PriceWheel` 是 4 位数字滚轮，配「元」标签；`toPrice()` 把 4 位数字拼成 `Int`（如 `0099 → 99`）。
- **隐患**：UI 侧本来就是整数「分」语义，却在边界 `toDouble()` 成 `Double` 直接落库。若 `99` 本意是 99 分（¥0.99），`toDouble()` 后变成 `99.0`，被下游当作 **99 元**——即 100 倍量级错误。
- **结论**：改造的第一步不是「换类型」，而是**先确认 UI 滚轮 4 位数字的真实语义（元 or 分）与现有库中 `originalPrice` 的真实量级**，否则迁移公式（`×100` 还是直接取整）无法定。见 §9 决策项 Q1。

## 3. 目标与非目标

**目标**
- 所有「金额」在内存与数据库中以**整数分（`Long`，单位：分）**存储与运算，消除浮点误差。
- 显示层在最后一步 `÷100` 格式化为「¥X.XX」；录入层在第一步 `×100` 转成分。
- 重构前后，现有 13 个单测的断言数值（以分为单位重新表达）**逐字节一致**。
- 多人分摊后「各人应付之和 == 总应付」「各人优惠之和 == 总优惠」，无 1 分漂移。

**非目标**
- 不引入任意精度小数（货币恒为 2 位小数，无需 `BigDecimal` 的通用精度）。
- 不改折扣算法本身的逻辑与近似边界（>16 件走贪心、≤16 件精确路径不变）。
- 不增加新的业务功能。

## 4. 方案对比与推荐

| 维度 | 方案 A：Long 分（推荐） | 方案 B：BigDecimal |
|------|------------------------|---------------------|
| 精度 | 整数运算，零误差 | 精确，但需显式指定舍入 |
| 性能 | 最优，无对象分配；对 `DiscountCalculator` 的 2ⁿ 热路径友好 | 每次运算新建对象，热路径有分配压力 |
| 与现有 UI 契合 | UI 已是 Int 分（`toPrice()`），天然对接 | 需额外转换层 |
| Kotlin 易用性 | 直白；除法需自管舍入 | API 啰嗦，易漏 `.toString()`/scale |
| 风险 | 分摊「凑整+余数再分配」需谨慎实现 | 舍入规则分散各处易漏 |

**推荐：方案 A（Long 分）**。
理由：① 性能最关键（折扣计算是 2ⁿ 枚举，Double→Long 反而更快且无 GC 压力）；② 与现有 `toPrice()` 整数分语义天然一致；③ 货币恒为 2 位小数，BigDecimal 的通用精度是过度设计。

**例外（不转分）**：`ownerFairness` / `fairnessScore` 是 0~1 的**公平性比率**，不是金额，保持 `Double`（或缩放为 0~1000 的整数比率），**切勿 ×100**。

## 5. 分摊凑整与余数再分配（最关键的设计点）

多人凑单按权重 `w`（Double 分数）分摊总优惠 `incrementalCents`：
```
part_i = incrementalCents * w_i      // 除法，必须定舍入
```
直接 `round(part_i)` 会导致 `Σ part_i ≠ incrementalCents`（差 1 分）。

**采用「最大余数法（largest remainder）」保证闭合**：
1. 计算每人精确份额 `exact_i = incrementalCents * w_i`（用 `Long` 高精度中间量，如先 `×1000` 保小数）。
2. 取下整 `floor_i = exact_i.toLong()`，累计 `Σ floor_i`，余数池 `remain = incrementalCents - Σ floor_i`。
3. 按 `exact_i - floor_i` 的小数部分从大到小，把 `remain` 个 1 分依次补给人，直到补完。
4. 断言 `Σ part_i == incrementalCents`，单测覆盖。

显示时每人 `part_i / 100.0` 格式化为「¥X.XX」。
